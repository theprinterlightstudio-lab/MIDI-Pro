Getting Started

Absolute Beginner Quick Start

Already know Touch Portal? Skip to Core Concepts. This section gets someone with zero experience to a working test in three steps.

What this plugin actually does

MIDI Pro turns Touch Portal buttons, sliders, and dials into a MIDI controller. Press a button, and MIDI Pro sends a real MIDI message out of your computer to your DAW, a hardware synth, or any other MIDI-listening software. It can also listen to MIDI coming in, and — its signature feature — automatically show you what was just sent, without needing your gear to echo anything back.

No hardware? No problem Install a free virtual MIDI cable and test everything on your computer alone: loopMIDI on Windows, or the built-in IAC Driver on macOS (Audio MIDI Setup → Window → Show MIDI Studio). Either one shows up in every MIDI Out / MIDI In dropdown in MIDI Pro and in your DAW.

Your first action, in five steps

Open a Touch Portal page and click a button to edit it.
In the Actions list, open the MIDI Pro category.
Drag MIDI Note onto the button.
Set MIDI Out to your virtual port or real device.
Press the button. If your DAW is listening on that port, a note comes in — the whole loop works.

Watch it react

Try MIDI Learn next — press a key on your real keyboard and watch MIDI Pro tell you exactly what it received. It's the fastest way to understand this plugin's whole design: it doesn't just send blindly, it can show you everything.

A Two-Minute MIDI Primer

Four terms come up everywhere in this plugin.

Channel 1–16
MIDI devices address 16 independent channels — like 16 separate radio frequencies. Most simple setups just use channel 1.
Note 0–127
Which musical note. 60 is Middle C — the pitch a piano key would generate.
Velocity 0–127
How hard the note was hit. 127 = hardest. Velocity 0 usually means "note released" in disguise.
CC / Control Change 0–127 / 0–127
The general-purpose knob message. The CC number identifies which control (CC 7 = Volume, CC 10 = Pan by convention). The value is how much.

Everything else in MIDI Pro is built from these four ideas.

The Actions

Every action lives under the MIDI Pro category in Touch Portal's action list. Seven actions total.

Mode behavior — read this first (Midi Note & CC share these six modes)

The names are not what you'd guess — "On/Off" is a one-shot pulse, "Momentary" is the true press-and-hold.

On / Off
Press fires ON and OFF immediately, back to back — a one-shot pulse. Use for drum hits, one-shot samples.
On only
Press sends only the ON velocity. Nothing on release.
Off only
Press sends only the OFF velocity.
Hold
Press sends ON. Release does nothing — turn it off some other way.
Momentary
Press sends ON, release sends OFF — a true held key, like a sustain pedal. What most people expect from “On/Off.”
Toggle
Each press alternates ON → OFF → ON. Release is ignored. Good for mute buttons that stay lit.
The mode people actually want, most often "Hold the button down, it plays; let go, it stops" is Momentary, not "On / Off."
MIDI Note NOTE ON / OFF
Sends a Note On / Note Off message — the message a piano key generates.

The Connectors — Sliders & Dials

You drag these instead of pressing them. Every connector comes in two visual flavors (Slider and Dial) that behave identically — pick whichever fits your layout.

CC Fader / Rotary Dial SLIDER + DIAL
The main way to control a continuous parameter — volume, filter cutoff, pan.
ModeAbsolute — position maps directly to a 0–127 CC value. Relative (3 variants) — acts as an endless encoder, sending ticks instead.
CC# / ChStandard CC target.
SensitivitySlow→Fast. Relative mode only — how fast ticks accelerate away from center.
CZ (Center Zone)Dead band around center where nothing is sent. Relative mode only.
CC Comp / Rotary CC Comp SLIDER + DIAL
For when “up” and “down” need to be two genuinely different messages, not just ±1 of the same one.
Up/Down CC#, Ch, Val, SensFully independent packets for each direction.
CZDead band around center, shared by both directions.
DZ (Dead Zone)Extra hysteresis buffer — stops jitter at the boundary from double-firing.
Pitch Bend Slider / Dial BOTH
A fader that continuously controls pitch bend instead of a CC. Center = no bend (8192).
Name / ChThe only two fields you'll touch beyond ports.
Program Change Slider / Dial BOTH
Drag across 0–100% and land on a specific patch number (0–127) — a quick patch browser.
Name / ChThe only two fields you'll touch beyond ports.
Which mode should you pick? If your DAW's fader jumps to match your slider position exactly — use Absolute. If it's an "infinite rotary" knob that only understands relative nudges — use a Relative variant (Binary Offset is the most universally supported; try that first).

MIDI Learn

MIDI Learn — Diagnose Anything

Solves the single most common beginner problem: "I don't know what my controller actually sends." Not a sender — a diagnostic listener.

How to use it

Drag MIDI Learn onto a button.
Set MIDI In to the port your hardware or DAW is connected through.
Press the button, then immediately play a note or move a knob on your real gear.
MIDI Pro captures the very next message and publishes it to the states below.

What you get back

StateExample value
state.learn.displayNOTE Ch1 N60 Vel127 — compact, perfect for a button label
state.learn.summaryMulti-line: Device, Type, Channel, Note/CC/Program, Velocity/Value, raw hex bytes
state.learn.deviceX-Touch Mini — which port it came from
state.learn.timestamp2026-07-04 14:32:07 — proves the capture is fresh
.type / .channel / .control / .value / .rawThe individual pieces, for your own display
Real use case A friend hands you an unlabeled controller. Put MIDI Learn on a button, show state.learn.summary on a second button (see Showing a State On Screen), and press every knob and pad once. Complete device map in five minutes.
Runtime Telemetry

Runtime Telemetry — Persistent States

MIDI Pro's signature feature: Touch Portal can show you, live, what was just sent — without your DAW or hardware sending anything back.

It only works if you fill in Name Name a control "Volume" and press/move it once, and MIDI Pro creates a small set of live states under that name automatically. Leave Name blank and nothing is created.

The states you get, per control type

You named a…Primary stateRangeAlso get
MIDI Note “Play”tx.note.play.stateon/offtx.note.play.velocity
CC / CC Fader “Volume”tx.cc.volume.value0–127tx.cc.volume.state (≥64)
Pitch Bend “Expression”tx.pb.expression.value0–16383tx.pb.expression.pct (−100…+100)
Program Change “Patch”tx.pc.patch.program0–127—
CC Relative/Comp “Jog”tx.dial.jog.value0–127, runningtx.dial.jog.direction

Every one of these also gets a .time companion showing exactly when it last fired (HH:MM:SS).

Shared position Name a connector the same as a button action — both called "Jog" — and they share the exact same live position. Turning a button and dragging a slider both called "Jog" moves one single number, not two.
Runtime Telemetry

Making a State Show Up on Screen

Read this carefully — it's the step beginners most often skip. Naming a control and pressing it does not automatically make its value appear anywhere. MIDI Pro only creates the state in the background. You still have to tell a specific button, on a specific event, to display it.

Touch Portal's native action: Update Button text…
Lives specifically under a button's On Event tab — not On Press.
Press/move your named control once first, so its state exists.
Edit the button where the value should appear — same button or a different one.
Switch to that button's On Event tab.
Add Touch Portal's built-in Update Button text… action.
Use the variable picker to insert your state — listed under MIDI Pro by the exact name you typed.
The easy shortcut Touch Portal has a built-in periodic refresh event (~500ms) that re-evaluates a button's text on its own, with no MIDI Pro event needed. Attach your state variable to that instead for a live number that "just works."
Sliders can't display their own feedback Touch Portal connectors (sliders/dials) have no On Event tab of their own — they're input-only widgets. The display always has to live on a separate button, text, or gauge element next to the slider, never the slider itself.
Runtime Telemetry

TX Events — For Automation, Not Display

One momentary trigger per MIDI type. Use these in Touch Portal's On Event system to make things happen automatically — switch a page, fire a notification, run another action — the instant any named control transmits.

EventFires when
MIDI Note SentAny named Note action sends
MIDI CC SentAny named CC action or CC-type connector sends
MIDI Pitch Bend SentAny named Pitch Bend action or slider/dial sends
MIDI Program Change SentAny named Program Change action or slider/dial sends
MIDI Dial ChangedAny named CC Relative action or Comp connector ticks

Shared context — tx.last.*

StateMeaning
tx.last.nameWhich control just fired, e.g. Volume
tx.last.typeNote / CC / Pitch Bend / Program Change / Dial
tx.last.valueWhat it sent
tx.last.portWhich MIDI port
tx.last.channelWhich channel
Do not use tx.last.* for display The next transmission from any control overwrites it. Built for the split second an event handler reads it — not for long-term display. For display, always use the per-name states instead.
Runtime Telemetry

Runtime States vs. TX Events

These solve two different problems. Once the distinction clicks, the rest of this plugin gets much easier to reason about.

Runtime StatesTX Events
Use whenYou already know the specific controlYou want to react to any transmission of that type
Readstx.note.play.state
tx.cc.volume.value		tx.last.name / .type / .value
Good forButton icons, LEDs, gauges, text, progress barsNotifications, logging, macros, generic workflows
SetupBind display element directly to the stateAttach to one of the five events, branch on tx.last.*
Why both exist Runtime States are for UI — no plugin Event required, this is the whole reason the TX Runtime State Publisher exists. TX Events are for automation — one generic handler that reacts to anything of a given type without knowing in advance which named control fired. Two different responsibilities, cleanly separated.

Real-World Scenarios

01
The Live Performer — A Volume Fader That Shows Itself
Confirm your master volume fader actually moved, without staring at your DAW.
Drag a CC Fader connector, Mode = Absolute, CC# = 7 (MIDI standard Volume), Name = Master.
On a nearby button's On Event tab, add Update Button text… pointed at tx.cc.master.value.
Every drag of the fader updates the readout — zero extra wiring in your DAW.
02
The Sample Trigger Wall
Eight buttons, each firing a different one-shot sample.
Eight MIDI Note actions, Mode = On/Off (one-shot pulse), one Note number per button (60, 62, 64…), same channel.
Map those note numbers in your sampler/DAW to eight sounds.
Optional: name each and add Update Button text… → tx.last.name on a shared indicator for an "on air" readout.
03
The Synth Patch Browser
Two buttons — Next / Previous Patch — walking through 128 presets.
One Program Change action, Mode = Increment (+1 on Hold), Name = Patch.
A second, Mode = Decrement (-1 on Hold), same Name (they share the live position).
On a display button's On Event tab, add Update Button text… → tx.pc.patch.program.
04
The Expression Pedal Replacement
No real expression pedal, but your patch has a filter sweep on CC 74.
CC Fader connector, Absolute mode, CC# = 74, Name = Filter.
On a gauge widget's On Event tab, add Update Button text… → tx.cc.filter.value.
05
The DJ-Style Filter Knob
An endless rotary feel using two adjacent buttons instead of a touch slider.
CC Relative (CW/CCW) action, CW/CCW Val set to your DAW's relative-CC mapping (Binary Offset's 65/63 is most common), Name = Jog.
On a display element's On Event tab, add Update Button text… → tx.dial.jog.value for a live visual as you spin it.

Troubleshooting

I named my control, pressed it, but nowhere shows the value.
Expected — naming only creates the state in the background. Add Touch Portal's Update Button text… action under some button's On Event tab and point it at that state yourself.
A MIDI In or MIDI Out dropdown is empty.
Make sure the device is connected (or your virtual port app is running) before opening the action's settings, then reopen it.
I pressed a named button but its state never showed up in the state picker.
States are created the first time a named control actually fires — press or move it at least once first.
My Toggle button doesn't seem to alternate correctly.
Toggle mode ignores button release entirely — only presses count. Make sure you're releasing between presses.
Nothing happens when I press a button at all.
Check that MIDI Out is actually set — an unset port silently does nothing rather than erroring, by design.
Values seem “off by one direction.”
For CC Relative / CC Comp, check which value your DAW expects for increase vs decrease. Use MIDI Learn on a working relative knob to see the exact byte pattern, then match it.
