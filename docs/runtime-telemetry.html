Runtime Telemetry

MIDI Pro's signature feature: Touch Portal can show you, live, what was just sent — without your DAW or hardware sending anything back.

It only works if you fill in Name Name a control "Volume" and press/move it once, and MIDI Pro creates a small set of live states under that name automatically. Leave Name blank and nothing is created.

Persistent States

One set per named control — this is what you bind dashboards, gauges, icons, and text to.

You named a…Primary stateRangeAlso get
MIDI Note “Play”tx.note.play.stateon/offtx.note.play.velocity
CC / CC Fader “Volume”tx.cc.volume.value0–127tx.cc.volume.state (≥64)
Pitch Bend “Expression”tx.pb.expression.value0–16383tx.pb.expression.pct (−100…+100)
Program Change “Patch”tx.pc.patch.program0–127—
CC Relative/Comp “Jog”tx.dial.jog.value0–127, runningtx.dial.jog.direction

Every one of these also gets a .time companion showing exactly when it last fired (HH:MM:SS).

Shared position Name a connector the same as a button action — both called "Jog" — and they share the exact same live position.

Making a State Show Up on Screen

Naming a control and pressing it does not automatically make its value appear anywhere. You still have to tell a specific button, on a specific event, to display it.

Touch Portal's native action: Update Button text…
Lives specifically under a button's On Event tab — not On Press.
Press/move your named control once first, so its state exists.
Edit the button where the value should appear — same button or a different one.
Switch to that button's On Event tab, add Update Button text….
Use the variable picker to insert your state, listed under MIDI Pro.
Sliders can't display their own feedback Touch Portal connectors have no On Event tab of their own. The display always lives on a separate button, text, or gauge element.

TX Events

One momentary trigger per MIDI type, for automation that needs to react to any transmission of a given kind without knowing in advance which named control fired.

EventFires when
MIDI Note SentAny named Note action sends
MIDI CC SentAny named CC action or CC-type connector sends
MIDI Pitch Bend SentAny named Pitch Bend action or slider/dial sends
MIDI Program Change SentAny named Program Change action or slider/dial sends
MIDI Dial ChangedAny named CC Relative action or Comp connector ticks

Shared context — tx.last.*

StateMeaning
tx.last.nameWhich control just fired, e.g. Volume
tx.last.typeNote / CC / Pitch Bend / Program Change / Dial
tx.last.valueWhat it sent
tx.last.portWhich MIDI port
tx.last.channelWhich channel
tx.last.note / tx.last.velocity / tx.last.stateNote-specific context
tx.last.ccCC-specific context
tx.last.programProgram Change-specific context
tx.last.pitchbendPitch Bend-specific context
tx.last.directionDial-specific context
Do not use tx.last.* for display The next transmission from any control overwrites it. For display, always use the per-name persistent states above instead.

Transport Feedback States

A separate, passive feature: 8 read-only states that reflect incoming Mackie Control Universal (MCU) style transport LED messages. This is not tied to Name/TX telemetry, and there's no action to trigger these yourself — they only change when a DAW sends the corresponding MCU note to a MIDI input port MIDI Pro is listening on.

StateMCU Note
state.transport.play94 (0x5E)
state.transport.stop93 (0x5D)
state.transport.record95 (0x5F)
state.transport.pausesynthesized, not a native MCU note
state.transport.loop86 (0x56)
state.transport.click89 (0x59)
state.transport.punchin87 (0x57)
state.transport.punchout88 (0x58)

"Pause" isn't a native MCU button — MIDI Pro synthesizes it when both Play and Stop report OFF shortly after Play was last ON, matching typical DAW UI behavior.
